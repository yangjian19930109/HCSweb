# 卡片图自动裁剪居中 — 经验总结

> 日期：2026-06-14
> 关联：[产品卡片比例修正方案](../产品卡片比例修正方案.md)、[DEVLOG](../DEVLOG.md#2026-06-14)

## 背景

产品卡片图需要将电机主体在正方形卡片中尽可能大地显示，且保留透明背景。管理后台上传时由 Canvas 自动处理。

## 问题演进（三轮迭代）

### 第一轮：白底 + 留白破坏图片

旧代码做了两件错事：

```javascript
// 错 1：白色填充 → PNG 透明通道全毁
ctx.fillStyle = '#ffffff';
ctx.fillRect(0, 0, canvas.width, canvas.height);

// 错 2：15% 留白 → 电机占比缩小
var padding = Math.round(Math.min(w, h) * 0.15);
canvas.width = w + padding * 2;
canvas.height = h + padding * 2;
ctx.drawImage(img, padding, padding, w, h);
```

后果：透明角变成不透明白 `(255,255,255,255)`，电机在画面中缩小 → `object-fit:cover` 在正方形卡片中进一步放大裁切。

**教训 1**：Canvas 初始状态全透明 `(0,0,0,0)`，**永远不要 fillRect**。想保留透明就什么都不填，`toBlob('image/png')` 天然保留 alpha。

### 第二轮：自动裁边但居中偏位

去掉白底和留白后，加上了自动扫描透明边界 + 裁切正方形：

```javascript
// 采样扫描非透明边界
var rmin, rmax, cmin, cmax;
for (var y = 0; y < h; y += stride) {
  for (var x = 0; x < w; x += stride) {
    if (alpha > 0) { /* 更新边界 */ }
  }
}

// 裁剪正方形 —— 这里有 bug
var centerY = Math.round((rmin + rmax) / 2);
var cropY = Math.max(0, centerY - Math.round(side / 2));
cropY = Math.min(cropY, h - side);  // ← 边界钳制
ctx.drawImage(srcCanvas, cropX, cropY, side, side, 0, 0, side, side);
```

**Bug 表现**：1031001 电机严重偏下，上留白 64%、下留白 0%。

**根因**：源图内容偏下时，`cropY = Math.min(centerY - side/2, h - side)` 被钳制到 `h - side`，正方形底边贴齐源图底边，内容被推到正方形顶部。

**教训 2**：**裁剪 + 边界钳制必然偏位**。内容靠近源图任何一边时，`Math.min/max` 会把正方形挤到对侧。这个方案在理论上有缺陷 — 只要内容不居中，输出就不居中。

### 第三轮（终版）：提取内容 + 居中绘制

不裁剪，改为从源图**提取内容区域，居中绘制到新画布**：

```javascript
// 计算正方形边长
var contentW = cmax - cmin + 1;
var contentH = rmax - rmin + 1;
var margin = Math.round(Math.max(contentW, contentH) * 0.10);
var side = Math.max(contentW, contentH) + margin * 2;

// 提取内容，居中绘制到新透明画布
var canvas = document.createElement('canvas');
canvas.width = side;
canvas.height = side;
var ctx = canvas.getContext('2d');
var dx = Math.round((side - contentW) / 2);
var dy = Math.round((side - contentH) / 2);
ctx.drawImage(
  srcCanvas,
  cmin, rmin,          // 源图内容区起点
  contentW, contentH,   // 源图内容区尺寸
  dx, dy,               // 目标画布位置（居中）
  contentW, contentH    // 目标画布尺寸
);
```

**效果**：无论源图内容在什么位置（偏上/偏下/偏左/偏右），输出始终完美居中。1031001 验证：上留白 965px，下留白 964px，偏移 = 0。

**教训 3**：**提取 + 合成 > 裁剪 + 钳制**。`drawImage` 的 9 参数重载（`sx, sy, sw, sh, dx, dy, dw, dh`）天然解决"源区 → 目标区"的映射，不需要手动计算钳制。

## 完整处理流水线

```
用户选择卡片图
  → FileReader 读取为 Image
  → 绘制到临时 Canvas，getImageData 获取像素
  → 采样扫描 alpha > 0 的边界（stride = min(w,h)/400）
  → 计算正方形边长 = max(contentW, contentH) × 1.10
  → 提取内容区，drawImage 居中绘制到新透明 Canvas
  → toBlob('image/png') 输出
```

## 其他踩坑

### 浏览器缓存 JS 导致"修复无效"

**现象**：改了 `admin.html` 的 `handleCardImgSelect()`，上传仍走旧逻辑。

**原因**：
1. 浏览器缓存旧 JS（serve.py 的 `_serve_admin()` 没设 Cache-Control 头）
2. Windows `SO_REUSEADDR` 允许多进程监听同一端口，旧进程残留

**解决**：
```bash
# 杀光所有 8080 进程
for pid in $(netstat -ano | grep ':8080' | grep LISTENING | awk '{print $5}'); do
  taskkill /F /PID $pid
done
# 重启服务 + 浏览器 Ctrl+F5 硬刷新
```

### 源图选错导致假性失败

**现象**：用户反复上传已被旧代码损坏的白底图片，每次 Canvas 正确保留它 → 看起来"没修好"。

**排查方法**：PIL 检查四角像素 alpha 值，`alpha=0` 才是透明源图。
