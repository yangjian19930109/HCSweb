# 图片拖拽不一致修复方案

> 2026-06-11，追查图片拖拽排序功能表现不一致的根因。

---

## 现象

管理后台编辑产品时，主图网格内部分图片可拖拽、部分不可拖拽，表现不一致：

| 产品 | 图片数 | 可拖拽 |
|------|--------|--------|
| 1030837 | 3 | 仅第1张 |
| 1030896 | 3 | 仅第1张 |
| 1031001 | 3 | 仅第2、3张（第1张不可） |
| 1061418 | 12 | 全部不可 |

---

## 根因

**`<img>` 的原生拖拽拦截了 `<div>` 的 HTML5 拖拽。**

浏览器对 `<img>` 元素有内置的原生拖拽行为（拖到桌面保存、拖到新标签页打开等）。当 `<div draggable="true">` 内部包含 `<img>` 时，浏览器优先触发图片的原生拖拽——`div` 的 `ondragstart` 事件根本不会被触发。

是否拦截取决于浏览器是否判定图片为"有效可拖图片"（`img.complete === true` 且 `naturalWidth > 0`），这受文件大小、缓存状态、加载时序影响，因此表现不一致。

### 定位

`backend/admin.html` 的 `renderImgGrid()` 函数中：

```javascript
var img = document.createElement('img');
img.src = item.src;
// ← 缺少：img.draggable = false;
div.appendChild(img);
```

---

## 修复

### 1. JS 层（必须）

`renderImgGrid()` 中，`img.src` 赋值后加一行：

```diff
 var img = document.createElement('img');
 img.src = item.src;
+img.draggable = false;  // 禁止原生图片拖拽，让 div 的 HTML5 拖拽接管
 div.appendChild(img);
```

**位置**：`backend/admin.html`，`renderImgGrid` 函数内，约第 372 行。

### 2. CSS 层（防御）

在 `<style>` 块的 `.img-grid-item img` 规则中追加：

```diff
 .img-grid-item img {
     width: 100%;
     height: 100%;
     object-fit: cover;
+    user-drag: none;
+    -webkit-user-drag: none;
 }
```

### 根因判定：Codex 分析更精确

| | 方案原始 | Codex | 复审 |
|---|---|---|---|
| 根因 | `<img>` 原生拖拽拦截 | Chromium 大图预览生成失败 | **Codex 更精确。** 如果仅原生拖拽导致，所有小图也应该受影响——但小图能拖 |
| 分界线 | 无数据 | ≤1872px 可拖，≥2686px 不可拖 | **数据清晰。** 12 张图的分界线与 `naturalWidth` 强相关 |
| 两者关系 | — | — | **互补。** 方案原始根因解释了"为什么部分图能拖"（小图无原生拦截），Codex 解释了"为什么部分图不能拖"（大图预览生成失败） |

实际机制可能是两层叠加：
1. 所有图片都有原生拖拽风险 → `img.draggable = false` 解决
2. 即使禁了原生拖拽，Chromium 仍需为 `<div>` 生成拖拽预览图 → 大分辨率图片导致预览生成失败 → `setDragImage` 解决

➡️ **合并方案的设计逻辑正确：两处修复互不冲突，各堵一个漏洞。**

---

### 修复项逐条确认

#### Fix #1：`img.draggable = false` ✅ 通过

```javascript
img.draggable = false;
```

- 阻止浏览器对 `<img>` 元素的默认拖拽行为
- 确保 `dragstart` 事件由 `<div>` 触发，而非被 `<img>` 截获
- 标准做法，无副作用

#### Fix #2：`setDragImage` ✅ 通过

```javascript
var d = new Image();
d.src = 'data:image/gif;base64,R0lGODlhAQABAIAAAAAAAP///yH5BAEAAAAALAAAAAABAAEAAAIBRAA7';
e.dataTransfer.setDragImage(d, 0, 0);
```

- 向 Chromium 提供一个 1×1 透明 GIF 作为拖拽预览图
- 绕过 Chromium 对大图的预览渲染（已知的 Chromium 限制：超大 `naturalWidth` 的图片在生成拖拽预览时可能消耗过多 GPU 资源导致静默失败）
- `(0, 0)` 偏移使预览图不可见，用户体验无影响
- Base64 字符串验证：`R0lGODlhAQABAIAAAAAAAP///yH5BAEAAAAALAAAAAABAAEAAAIBRAA7` → 合法的 1×1 透明 GIF ✅

#### Fix #3：`user-drag: none` ✅ 通过

```css
user-drag: none;
-webkit-user-drag: none;
```

- CSS 层防御，阻止浏览器将 `<img>` 识别为可拖拽对象
- 与 Fix #1 功能重叠但互不冲突（JS 层 + CSS 层双重保险）
- 标准属性，无副作用

| 风险 | 评估 |

| `setDragImage` 空预览是否影响用户体验 | ✅ 不影响。拖拽时显示的幽灵图变为透明，但 `.img-grid-item` 本身的 `opacity`/边框高亮仍清晰指示拖拽状态 |
